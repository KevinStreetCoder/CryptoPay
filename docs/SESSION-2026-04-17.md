## Problems fixed

### 1. "Logged out after a day — new device" UX bug
Two root causes:
- `REFRESH_TOKEN_LIFETIME = 1 day` → forced daily re-login.
- `LoginView` challenged on **either** new device_id **or** IP change. Mobile IPs rotate
  constantly (WiFi ↔ cellular), so returning users were flagged as "new device" even
  on the same phone.

**Fix:**
- Refresh TTL → 30 days (rotation + blacklist still on, so a stolen token is
  invalidated on the next legitimate refresh).
- IP-change branch only triggers when the device is **not** already trusted.
- OTP response now includes `challenge_reasons[]` and a reason-specific `message`
  ("New device detected" vs "Sign-in from a new location") — mobile shows the
  accurate copy directly.

**Files:** `backend/config/settings/base.py:336-345`, `backend/apps/accounts/views.py:348-427`.

**Tests:** `LoginSecurityChallengeTest` x 3 (`apps/accounts/tests.py`). All green.

### 2. eSMS Africa verification + observability
eSMS sent an "everything is restored" email but their API was still silently
dropping messages for some numbers.

**Fix:**
- `backend/apps/core/email.py` — structured `sms.dispatch` log entries
  (provider, masked recipient, status_code, latency_ms, fallback_triggered,
  response_snippet) on every attempt. Catches silent provider failures
  immediately.
- New `/admin/health/sms/` endpoint (staff-only) — POST `{to, message}` to
  send a real test SMS on demand.

**Verified:** test to +254701961618 delivered successfully. Payload: eSMS returned
an error, Africa's Talking fallback delivered. Infrastructure is resilient;
eSMS itself is still flaky for some MSISDNs despite their apology.

### 3. Google Sign-In `redirect_uri_mismatch`
User hit "Access blocked: invalid request — Error 400: redirect_uri_mismatch".

**Root cause:** The OAuth client in Google Cloud Console didn't have the
production callback URL registered.

**Fix (user-side, done):** added these 4 URIs to
`797403635202-t2i871kkg2qfsoq19appg47c6ud0c6ac` Web client:
- `https://cpay.co.ke`
- `https://cpay.co.ke/auth/google/callback`
- `http://localhost:8081/auth/google/callback`
- `https://auth.expo.io/@kevinkareithi/cryptopay`

### 4. Google OAuth "Unmatched Route" after redirect
After Google redirected back to `/auth/google/callback#id_token=...`, the
Expo Router app had no file at that path and rendered "Unmatched Route".

**Fix:** new file `mobile/app/auth/google/callback.tsx` that calls
`WebBrowser.maybeCompleteAuthSession()`, parses the `id_token` from the URL
fragment, exchanges it via the existing `googleLogin()` action, then routes
to `/(tabs)` or the profile-completion screen.

### 5. Over-engineered loading spinner
`LoadingScreen.tsx` was running 6 parallel animations: logo pulse (1.2s loop),
rotating spinner (2s), two expanding pulse rings with staggered 1s delay,
fade-in, slide-up, plus an ambient blur glow. Users reported it felt like
4–6 seconds of loading even on fast connections.

**Fix:** rewritten with **one** subtle indeterminate shimmer bar + a 260ms
fade-in. Logo and wordmark are stable. Inspired by how Wise, Revolut, and
Cash App handle cold-load. Now reads as ~200 ms of polish instead of the
old pulsing-ring theatre.

### 6. WCAG AA text contrast
`textSecondary: #8899AA` was ~3.5:1 on the dark background — below the 4.5:1
AA requirement.

**Fix:** `theme.ts` — `textSecondary` → `#A8BBCC`, `textMuted` → `#6B8299`.
Both now ≥ 4.5:1.

## New: Push-notification 2FA (Apple/GitHub/Revolut-style tap-to-approve)

### What it is
When a login triggers a security challenge (new device / new IP), instead of
making the user read and type a 6-digit SMS code, we send a push notification
to their **other trusted phone** that says "Approve sign-in from [device]?"
with Approve / Deny buttons. One tap completes login on the original device.

Benefits: no SMS cost, no SIM-swap attack surface, no digit-typing friction,
instant UX. Same pattern Google / GitHub / Revolut / Binance use.

### Architecture
- Redis-backed `LoginChallenge` (5-min TTL, 16-byte url-safe secret IDs,
  single-use). No DB migration needed.
- Expo push dispatched via the existing `send_push_notification()` — deep link
  `cryptopay://auth/approve-login/<challenge_id>` with requesting device + IP
  in the payload.
- Three endpoints (all live on production):
  - `GET  /api/v1/auth/challenge/<id>/status/` — polled by the login screen;
    unauthenticated because the challenge_id itself is the capability.
  - `POST /api/v1/auth/challenge/<id>/approve/` — authenticated; only the
    targeted user can approve.
  - `POST /api/v1/auth/challenge/<id>/deny/` — authenticated.
- `LoginView` dispatches **both** channels when the user has a PushToken;
  SMS OTP fires regardless as the fallback. The request body now accepts
  `challenge_id` — if it's approved, tokens are issued without any OTP.

### Files
- `backend/apps/accounts/push_challenge.py` — module (170 lines).
- `backend/apps/accounts/views.py` — `PushChallengeStatusView`,
  `PushChallengeApproveView`, `PushChallengeDenyView`, and the LoginView
  integration.
- `backend/apps/accounts/urls.py` — three new routes.

### Mobile-side (shipped tonight)
- `mobile/app/auth/approve-login/[id].tsx` — deep-link target. Shows the
  requesting device + IP, Approve / Deny buttons, confirmation states.
- `mobile/app/auth/login.tsx` — polls `/challenge/<id>/status/` every 3 s when
  the server dispatched a push challenge. Auto-completes login on `approved`.
- Blue banner "Check your other device" shown above the OTP input whenever
  a push was dispatched — but only when `push_challenge` is in the response,
  so single-device users never see it.

### Safety: nobody gets locked out

| User has... | Behaviour |
|---|---|
| Only the device trying to sign in | `create_challenge()` sees zero PushTokens, returns `None`, no push is sent. SMS OTP goes out as normal. UI shows the OTP keypad only — no banner. |
| Another trusted device with the app installed | Both channels fire simultaneously. Whichever the user responds to first wins. |
| Push fails (Expo down, token invalid) | Exception caught and logged, request proceeds — SMS always succeeds. |
| Push succeeds but user ignores it | 5-min timeout. SMS OTP still valid for its own TTL — user can type it anytime. |
| User denies on phone 2 | OTP input stays available in case the denial was a misclick. |

The SMS path is strictly untouched and always available. Push is additive.
